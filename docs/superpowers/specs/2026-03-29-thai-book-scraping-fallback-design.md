# AI Enrichment Fallback — Design Spec

**Date:** 2026-03-29
**Status:** Implemented

## Problem

Many books (especially Thai-language) are not indexed by Google Books. When a user scans such a book cover, the AI correctly identifies the title and author, but Google Books returns no match — resulting in a degraded "AI ONLY" experience with no metadata.

## Solution

When Google Books has no match, make a second Workers AI call to enrich the result with metadata (publisher, description, page count, categories). This works for any language, not just Thai.

### Why AI enrichment over web scraping

Originally planned to scrape Thai book retailer websites (Kinokuniya, Naiin, SE-ED) via Cloudflare Browser Rendering `/scrape`. Pivoted because:

- **Cost:** Workers AI is free tier vs Browser Rendering billing
- **Speed:** ~1-2s vs 4-20s for 2-6 scrape calls
- **Maintenance:** No fragile CSS selectors that break when retailer sites change
- **Simplicity:** Single service file vs scrape client + 3 source parsers + orchestrator

### Trade-offs

- LLM may hallucinate metadata (wrong publisher, made-up page count)
- Knowledge cutoff — very new or obscure books may not be in training data
- No thumbnail or infoLink (can't be generated by AI)

The `source: 'ai_enriched'` badge signals to users that metadata confidence is lower than a Google Books match.

## Architecture

### Enrichment Flow

```
User uploads cover
  → AI identifies title, author, AND language
  → Check KV cache → hit? return cached result
  → Try Google Books
    → match? → cache + return (source: "google_books")
    → no match? → AI enrichment call
      → success? → cache + return (source: "ai_enriched")
      → failure? → return (source: "ai_vision", title + author only)
```

### AI Prompt Change

The vision prompt now returns a `language` field:

```
Reply ONLY with JSON: {"title":"...","author":"...","language":"..."}
where language is an ISO 639-1 code like "en", "th", "ja".
```

`BookIdentification` type:

```typescript
export interface BookIdentification {
  title: string;
  author: string;
  language: string;
}
```

The `language` field is stored for future use (e.g., language-specific enrichment strategies) but currently all languages follow the same flow: Google Books → AI enrichment fallback.

### AI Enrichment Service (`ai-enrich.ts`)

Makes a second Workers AI call with the identified title/author:

```typescript
export async function enrichWithAi(
  ai: Ai,
  identification: BookIdentification
): Promise<BookMetadata>
```

The AI is prompted to return metadata JSON:

```
You are a book metadata expert. I identified a book from its cover:
Title: "..."  Author: "..."  Language: "..."

Reply ONLY with JSON: {"publisher":"...","publishedDate":"YYYY","pageCount":123,"categories":"...","description":"..."}
```

- Title and author are preserved from the original identification (not overwritten by AI)
- `infoLink` and `thumbnail` are always `null` (can't be generated)
- Returns `source: 'ai_enriched'` on success, `source: 'ai_vision'` on failure

### Response Shape

```typescript
export type BookSource = 'google_books' | 'ai_vision' | 'ai_enriched';
```

### Frontend Changes

Badge text based on source:
- `google_books` → "HIGH MATCH" (gold)
- `ai_enriched` → "AI ENRICHED" (blue)
- `ai_vision` → "AI ONLY" (grey, reduced UI)

`ai_enriched` results show the full metadata grid (publisher, date, pages, category, description) but no cover comparison or external link.

### Result Caching

Cache enrichment results in Workers KV to avoid repeated AI calls.

**Cache key:** normalized `{title}:{author}` lowercased and trimmed (e.g. `"เด็กหอ:ปราบดา หยุ่น"`).

**Cache value:** the full `BookMetadata` JSON response.

**TTL:** 7 days.

Caching applies to both `google_books` and `ai_enriched` results. `ai_vision` results (no enrichment) are **not** cached — the AI model may improve or Google Books may index the book later.

**Env binding:**

```typescript
export interface Env {
  AI: Ai;
  GOOGLE_BOOKS_API_KEY?: string | undefined;
  ALLOWED_ORIGIN: string;
  BOOK_CACHE?: KVNamespace | undefined;
}
```

### Error Handling

- If AI enrichment fails (timeout, parse error, garbage response), return `ai_vision` fallback
- Failures are silent — no error returned to the user
- Cache operations are best-effort (errors silently caught)

### Cost & Performance

- Workers AI calls are free tier (`@cf/meta/llama-3.2-11b-vision-instruct`, 10k neurons/day)
- **Best case:** 1 Google Books fetch (~200ms) → cache hit on repeat
- **Typical fallback:** 1 Google Books fetch + 1 AI call (~1-2s)
- **Full miss:** 1 Google Books fetch + 1 failed AI call (~1-2s) → `ai_vision`

### Testing

- 37 tests across 5 test files
- AI enrichment: 6 tests (success, code fences, partial metadata, garbage, throw, preserves title/author)
- Cache: 9 tests (key normalization, hit/miss, undefined KV, errors)
- AI identification: 7 tests (language detection, Thai, defaults, regex fallback)
- Google Books: 7 tests (full match, fallback returns `ai_vision`, etc.)
- Index integration: 8 tests (routing, enrichment fallback, CORS, rate limit)
